# DDR1 수동 송출 로그 파일 포맷 스펙

## 개요

- **파일 확장자**: `.Log`
- **파일명 예시**: `ddr1 송출내역.Log`
- **저장 주기**: 매일 1개 파일 생성
- **인코딩**: UTF-8
- **형식**: 타임스탬프 기반 텍스트 로그

---

## 기본 라인 구조

```
HH:MM:SS.mmm : [DDR1] <Direction> : <Event>
```

| 위치 | 예시 | 설명 |
|------|------|------|
| 타임스탬프 | `00:34:15.604` | HH:MM:SS.밀리초 |
| 장비명 | `[DDR1]` | 고정값 |
| 방향 | `I` / `O` | I=입력(명령), O=출력(상태) |
| 이벤트 내용 | `Play CLIP : N270944` | 이벤트 유형별 상이 |

---

## 이벤트 유형

### 입력 이벤트 (`I` 방향)

| 이벤트 패턴 | 예시 | 설명 |
|------------|------|------|
| `Remote Play` | `00:34:15.604 : [DDR1] I : Remote Play` | 원격 재생 명령 수신 |
| `Play Click` | `00:34:15.604 : [DDR1] I : Play Click` | 재생 버튼 클릭 |
| `Play` | `00:34:15.604 : [DDR1] I : Play` | 재생 실행 |
| **`Play CLIP : N######`** | `00:34:15.620 : [DDR1] I : Play CLIP : N270944` | **실제 클립 재생 시작** ← 핵심 이벤트 |

### 출력 이벤트 (`O` 방향)

| 이벤트 패턴 | 예시 | 설명 |
|------------|------|------|
| **`OnAir ID : N######, SOM : ..., EOM ..., Tc : ..., RemTime : ...`** | 아래 참조 | **현재 온에어 클립 상태** |
| `Load ID : N######, SOM : ..., Dur ...` | `00:34:21.822 : [DDR1] O : Load ID : N280892, SOM : 00:00:00:00, Dur 00:01:15:00` | 다음 클립 큐 로드 |
| `Cue Cmd > OK` | | 큐 명령 확인 |
| `Cue Done Ok!` | | 큐 완료 |
| `Play Ok => giOnAirTime : 12  iRemTime : 12` | | 재생 성공 확인 |
| `Play Clip Change Tc : ..., State : ...` | | 클립 전환 |
| `3Sec Re Load ID : ..., State : ... > Skip` | | 3초 재로드 (Skip이면 무시) |
| `Play Wait... OnAir ID : ...` | | 재생 대기 중 상태 |

## 핵심 이벤트 상세

### Play CLIP 이벤트 (송출 시작 기록)

```
00:34:15.620 : [DDR1] I : Play CLIP : N270944
```

- 수동 송출이 실제로 시작된 시점
- 클립 ID 형식: `N` + 6자리 숫자
- **파싱 정규식**: `^(\d{2}:\d{2}:\d{2}\.\d{3}) : \[DDR1\] I : Play CLIP : (N\d+)$`

### OnAir ID 이벤트 (온에어 상태)

```
00:34:21.822 : [DDR1] O : OnAir ID : N270944,  SOM : 00:00:00:00, EOM 00:00:09:29 , Tc : 00:00:05:29,  RemTime : 120
```

| 파라미터 | 예시 값 | 설명 |
|---------|---------|------|
| `OnAir ID` | `N270944` | 현재 온에어 클립 ID |
| `SOM` | `00:00:00:00` | Start Of Material (시작점) HH:MM:SS:FF |
| `EOM` | `00:00:09:29` | End Of Material (종료점) HH:MM:SS:FF |
| `Tc` | `00:00:05:29` | 현재 타임코드 HH:MM:SS:FF |
| `RemTime` | `120` | 남은 시간 (프레임 수) |

- **파싱 정규식**: `^(\d{2}:\d{2}:\d{2}\.\d{3}) : \[DDR1\] O : OnAir ID : (N\d+),\s+SOM : ([\d:]+), EOM ([\d:]+) , Tc : ([\d:]+),\s+RemTime : (\d+)$`

## 송출 이벤트 추출 로직 ⚠️ 폐기됨 (2026-06)

> 아래 "방법 1/2"는 로그 전체에서 독립적인 이벤트를 찾는 과거 방식이며, 실데이터 검증 결과
> 신뢰할 수 없어 더 이상 사용하지 않는다. 자세한 사유는 아래 "레거시 방식 폐기 사유" 참조.
> 현재는 APST `MM=DDR1` 앵커 기반 구간 추출만 사용한다 (아래 "수동 송출(DDR1) 구간 추출" 섹션).

### 방법 1: Play CLIP 이벤트 기준 (폐기)
```
"Play CLIP : N######" 라인 → 해당 클립이 수동 송출된 것으로 판단
```
- 예시 파일 기준: 39개의 Play CLIP 이벤트

### 방법 2: OnAir ID 이벤트 기준 (폐기)
```
"OnAir ID : N######" 라인 → 해당 클립이 온에어 상태
RemTime이 특정 임계값(예: 120) 이상인 첫 번째 OnAir 이벤트 = 클립 시작
```
- 예시 파일 기준: 305개의 OnAir ID 이벤트 (중복 상태 업데이트 포함)

### 레거시 방식 폐기 사유
1. **오탐**: 운영자가 화면에서 단발성으로 미리보기/테스트 재생한 클립도 `Play CLIP` 라인을
   남겨, 실제 송출이 아닌데도 진짜 송출로 기록됨
   - 예: 2026-05-01 07:33:37 `N280388`(우리아이치과(틀니)) — Play CLIP만 있고 OnAir 확인 자체가 없음(클릭만 하고 실제 재생까지 가지 않음)
   - 예: 2026-05-01 08:25:51 `N280709`(518재단) — 자기 자신만 OnAir로 확인되고 바로 정지(STOP), 다음 소재로 이어지지 않음
2. **중복**: "다음 소재로 이어지는 연속성"으로 단발성 여부를 검증해도, DDR1 장비가
   (`MM=DDR1`로 표시되지 않은) 정상 자동 송출 SB 띠까지 함께 기록하고 있어, apst.db에 이미
   저장된 자동 송출 내역과 거의 동일한 시각의 중복 데이터가 다시 적재됨

→ 수동 송출 여부를 신뢰성 있게 판별할 수 있는 유일한 기준은 APST가 명시적으로 표시하는
`MM=DDR1`뿐이다. 모든 수동 송출 내역은 그 앵커를 기준으로만 추출한다.

## 시간 처리

로그 파일에는 날짜 정보가 없고 시간만 존재:
- 파일명 또는 생성일자에서 날짜 추출 필요
- `00:00:00` ~ `23:59:59` 범위로 당일 날짜 적용
- 자정 전후(23:XX → 00:XX) 처리 시 날짜 경계 주의

## ✅ 수동 송출(DDR1) 구간 추출 (신규, 2026-06 시간 산출 방식 수정)

`Play CLIP : N######` 이벤트는 **SB 띠를 시작하는 이어서 소재에만** 찍힌다 (예시 파일 기준 실제 확인:
39개의 Play CLIP 이벤트 모두 각 SB 띠의 이어서 소재 clip_id와 일치). 이어서 다음에 오는 캠페인/ID/광고
소재들은 별도의 `Play CLIP` 라인 없이, `O : Play Clip Change Tc : ...` 라인의 타임스탬프가 실제
전환(송출 시작) 시점이다. 이 라인에는 clip_id가 없으므로, 그 직전에 나온
`O : Load ID : <clip_id>` 또는 `O : 3Sec Re Load ID : <clip_id>`에서 clip_id를 가져온다.

```
00:34:15.620 : [DDR1] I : Play CLIP : N270944           ← 이어서(체인 시작)
00:34:21.822 : [DDR1] O : OnAir ID : N270944, ...        ← 이어서 자신 확인 (시작 시간 아님)
00:34:21.822 : [DDR1] O : Load ID : N280892, ...          ← 다음 소재 큐에 등록 (clip_id 기억)
00:34:25.835 : [DDR1] O : Play Clip Change Tc : ...       ← N280892로 실제 전환된 시점 = 진짜 시작 시간
```

⚠️ **`OnAir ID` 라인은 실제 시작 시간으로 쓰면 안 된다** — 그 소재가 거의 다 재생된 뒤(보통 재생
종료 16~20초 전, 다음 소재 Load와 동시)에야 찍히는 상태 확인 라인이라 실제 시작보다 훨씬 늦다.
과거 이 라인 시간을 시작 시간으로 사용한 결과 (1) 모든 소재의 시작 시간이 실제보다 16~20초씩
늦게 기록되고, (2) SB 띠 종료 경계(`end_time`) 부근의 마지막 소재가 경계 비교 시점이 늦어져
통째로 누락되는 버그가 있었음 (2026-05-01 사례로 확인·수정).

APST에서 이 SB 띠의 이어서 소재 주장비명(`MM`)이 `DDR1`이면 수동 송출로 판단하고, 위 `Play Clip
Change` 시퀀스에서 소재를 시간 순서대로 추출한다. 마지막 소재는 다음 `Play Clip Change`가 없을 수
있음(운영자가 중간에 정지하는 경우 등) — 이미 전환이 확인된 마지막 소재까지만 인정한다.
파싱 함수: `ddr1_parser.extract_manual_clips()` (시작 시간 동기화 오차 ±5초 허용).
